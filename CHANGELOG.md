### Added
- **Transaction linking + portfolio-holding UX batch (2026-04-24).** Targeted follow-ups on the WealthPosition ZIP import surfacing as edge-case gaps.

  **Multi-leg transaction linking.** New nullable `transactions.link_id text` column + partial index on `link_id IS NOT NULL` ([scripts/migrate-transactions-link-id.sql](scripts/migrate-transactions-link-id.sql), applied prod/staging/dev). Every multi-leg `#SPLIT#` group that the WP ZIP transform emits (transfer set, same-account currency conversion, position liquidation, stock buy with cash + position legs) gets one `randomUUID()` `linkId` carried on every emitted row. UI: the edit dialog gains a blue "Linked transaction(s)" section listing siblings by date + account + amount when `link_id` is set (fetches `/api/transactions/linked?linkId=…&excludeId=…` — in-memory exact-match filter, user-scoped, inherits the same soft-DEK policy as `GET /api/transactions`). A small `Link2` icon in the category column of the table flags linked rows. Clicking a sibling jumps to its edit view; if it's on another page, the list filters to its date.

  **Three WP ZIP transform edge cases closed.** The single-walk through `emitGroup`'s holding-children branch used to skip the parent leg whenever the parent itself was a holding (`parentHoldingRef` set). That path was always-emit for cash parents but drop-parent for holding parents, and the consequences were silent:
  - **Aug-11-style same-account currency conversions** (parent `Joint - USD`, child `Joint - CAD`, both on IBKR Joint) emitted only the child with `category: undefined` — user saw one side land on IBKR Joint with a null category. Fix: always emit parent, use `holdingForAccountName(parent.account)` so the dual-natured account tags its own holding name as `portfolio_holding`, and tag both legs with the user's Transfer category.
  - **Jul-31-style liquidations** (sell 40 shares of `Joint - Dev Asia ex Japan - A` into `Joint - USD` cash) emitted only the cash leg — the aggregator never saw `-40 shares` so the position stayed at its pre-sell quantity. Fix: the parent sell leg now emits on its brokerage with `portfolio_holding = parent holding name`, `quantity < 0`, and the Transfer category.
  - **Aug-14-style cross-account transfers** (Joint - CAD → RBC Checking) worked but the two txs weren't linked. Fix comes from the `linkId` threading above — same group-level uuid on both.

  **Dividend source-holding as a tag, not a share move.** A WP row like `Joint - USD, Dividends, CAD, 20, 14.29, Joint - All W - D` encodes **two events in one row**: Joint - USD received $20 CAD / +14.29 USD qty, and that dividend was *generated by* the Joint - All W - D position. The previous transform stuffed `Joint - All W - D` into `tx.portfolio_holding`, so the aggregator credited +14.29 shares to Joint - All W - D (zeroing its cost basis) while Joint - USD's balance was never touched. The right read (confirmed against WP's UI): the balance change lands on the row's Account, and the "Portfolio holding" column is a reporting attribution. Both `emitSingleRowTx` (dual-natured account case) and `emitHoldingSideTx` (Portfolio-only case) now detect `csv.portfolio_holding != account.name` and append `source:<holding-name>` to `tags`, keeping the balance + qty on the account. Clickable tag chips + deep-linkable `?tag=source:X` filter let the user audit "all dividends from this holding" without a report script.

  **Portfolio holding edit + delete.** Each row in the `/portfolio` holdings table gains an **Edit** button in its expanded panel (next to "View transactions"). Dialog lets the user change name / symbol / currency / is-crypto flag / note, with a two-step confirm on delete. New `PUT /api/portfolio` (updates via user-scoped ownership pre-check) and `DELETE /api/portfolio?id=…` (returns 404 when nothing deletes, avoiding silent no-ops). The rename caveat is called out inline — `tx.portfolio_holding` is a plaintext (now encrypted) string not a FK, so renaming here doesn't cascade to existing transactions. Follow-up task spawned to switch to an integer FK so renames become free; tracked separately.

  **Portfolio deep-link to transactions.** Holding row expansion → **"View transactions →"** opens `/transactions?portfolioHolding=<name>&accountId=<id>`. `GET /api/transactions` now accepts `portfolioHolding` and `tag` query params — both are in-memory exact-match filters applied after `decryptTxRows` (SQL `LIKE` doesn't work on AES-GCM ciphertext, and substring match would false-match related tags). When either is set the server over-fetches `limit=1000 offset=0`, filters in memory, then re-applies the client's pagination to the filtered result. Transactions page reads URL params at mount (inside a new `<Suspense>` wrapper for `useSearchParams`), renders pill chips for active post-decrypt filters, and now surfaces a **Quantity** column between Note and Amount so investment legs show their share counts inline.

  **Portfolio-overview DEK-miss defense.** `GET /api/portfolio/overview` wrapped `decryptField` in `try/catch` and skips any row whose decrypted key still starts with `v1:` — post-restart DEK-cache misses previously rendered every unique AES-GCM IV as its own "auto-detected" orphan holding (1,000+ `v1:…` rows for a 6 000-tx user). The skipped count surfaces as `undecryptedTxCount`; the Portfolio page renders an amber "Sign in again to unlock" banner when non-zero so the user knows exactly what to do.

  **Case-insensitive orchestrator dedup + Stream D Phase 3 compat.** Both [zip-orchestrator.ts](src/lib/external-import/zip-orchestrator.ts) and [orchestrator.ts](src/lib/external-import/orchestrator.ts) now maintain two dedup indexes per table — one keyed by `trim().toLowerCase()` plaintext (legacy rows) and one keyed by the `name_lookup` HMAC (Stream D Phase 3 rows whose plaintext is nulled). Before this, case-sensitive plaintext-only lookup missed two real-world collisions: a single CSV containing both `Balance adjustments` and `Balance Adjustments` (same HMAC, unique-index collision on the second INSERT) and a prod user whose `Entertainment` category was Phase-3 encrypted (null plaintext, non-null `name_lookup`) so the orchestrator didn't see it in the Map and tried to auto-create a duplicate. `nameKey()` is also null-safe now — rows with null plaintext don't crash the map build.

- **WealthPosition migration connector — pluggable third-party import (2026-04-24).** First external-service connector, shipped as a new workspace package [`@finlynq/import-connectors`](packages/import-connectors/) (private, npm-publishable once a second connector validates the API). Lives at [/import → Connect a Service](src/app/(app)/import/components/connector-tab.tsx). Ships with **two paths**:

  **Primary: ZIP export upload.** User downloads WealthPosition → Profile → Export (a 4-CSV bundle: Accounts.csv / Categories.csv / Portfolio.csv / Transactions.csv) and uploads the ZIP. Server extracts via `jszip` (no native deps), parses via a hand-rolled RFC-4180 CSV reader ([csv.ts](packages/import-connectors/src/wealthposition/csv.ts)), and transforms into Finlynq's `RawTransaction[]` + `transaction_splits[]` + `portfolio_holdings[]`. Routes: [`/api/import/connectors/wealthposition/zip-{probe,preview,execute}`](src/app/api/import/connectors/wealthposition/). Instant (no rate limits) and handles 100% of transaction shapes. Portfolio.csv is the key: it's the authoritative holding → brokerage + ticker symbol mapping, which WP's API never exposes. Verified against a real 6,596-row export → 5,817 flat txs + 111 split parents + 60 portfolio_holdings rows + 0 transform errors.

  **Secondary: API key.** Live [WP API](https://www.wealthposition.com/api/v0.1) access, kept for balance reconciliation against `/account_balances`. Key is AES-GCM-encrypted under the user's DEK and persisted in `settings` under `connector:wealthposition:credentials`. Rate-limited client (1 req/s) with SHA-keyed global bucket on `globalThis` so state survives across HTTP requests in the same process; automatic retry with jittered backoff on `RATE_LIMIT_ERROR` / HTTP 429. Original plan had the API as the primary import path — pivoted to ZIP after discovering the API's `/accounts` flattens every position (Bitcoin, Ethereum, every stock ticker) as a separate top-level account with no way to tie them back to their brokerage.

  **Transform rules (ZIP path).** The CSV uses a `#SPLIT#` sentinel to preserve multi-leg transactions (parent row has `account=<real>` + `categorization=#SPLIT#`, consecutive children have `account=#SPLIT#`). Walker groups parent + consecutive children, classifies each child as category / account / holding, and emits:
    - **Simple 1A + 1C** → 1 Finlynq flat tx (4,204 rows, 63.5%).
    - **Cash + #SPLIT# + N categories** → 1 parent + N rows in `transaction_splits` (paycheck splits).
    - **Cash + #SPLIT# + N holdings** → parent cash leg preserved on the funding account + N position legs routed via `Portfolio.csv` to each holding's own brokerage (so a stock buy funded from RBC lands the cash leg on RBC and the shares on Questrade - RESP, each with `portfolioHolding = holding_name` + `quantity`).
    - **Cash + #SPLIT# + N accounts** → transfer set (parent + N flat txs, all tagged with the Transfer category).
    - **Orphan #SPLIT# rows** → surfaced as transform errors (shown in the preview dialog, not dropped silently).

  **Mapping + reconciliation.** Probe returns Finlynq's live accounts + categories so the mapping dialog can render names (not opaque ids); stale overrides (pointing at accounts deleted outside the import flow) are filtered client-side before opening the dialog and re-validated server-side in `materializeZipMapping`. Reconciliation endpoint fetches WP's `/accounts` once to build a `name→UUID` bridge, so CSV synthetic ids (`csv:acct:<name>`) match WP's real UUIDs from `/account_balances`. After a successful import, reconciliation runs inline when an API key is on file — per-account WP-vs-Finlynq diff table surfaces in the summary card, with an "Add opening balance" affordance for constant-offset mismatches.

- **Archive or delete accounts (2026-04-24).** The Accounts page's edit dialog now has an Account actions footer with **Archive / Unarchive** and **Delete**. Archive is a soft hide — the account drops out of balances, net-worth totals, and every account picker (transactions, loans, goals, subscriptions, import, onboarding) while keeping history intact. A "Show archived" toggle on the page header brings archived rows back into view at 60% opacity with an Archived badge so they can still be edited or unarchived; totals always exclude archived regardless of the toggle. Delete is permanent and guarded: the DELETE handler catches Postgres `foreign_key_violation` (SQLSTATE `23503`) and returns 409 with "Archive it instead, or remove the related records first" when any of the 8 referencing tables (transactions, portfolio_holdings, loans, subscriptions, goals, snapshots, transaction_splits, recurring_transactions) still points at the account. New column `accounts.archived boolean NOT NULL DEFAULT false`, migration at [scripts/migrate-accounts-archived.sql](scripts/migrate-accounts-archived.sql) (applied to prod; staging + dev pending). `getAccounts()` and `getAccountBalances()` gained an `{ includeArchived }` option that defaults to excluding archived — every current caller gets the right behavior without opt-in. `GET /api/accounts?includeArchived=1` and `GET /api/dashboard?includeArchived=1` are the opt-in paths used by the toggle. `PUT /api/accounts` accepts `archived: boolean`; `DELETE /api/accounts?id=…` is new.
- **Email import via Resend Inbound with staging review queue (2026-04-23, commits [8dbf0db](https://github.com/finlynq/finlynq/commit/8dbf0db) → [a1e1b51](https://github.com/finlynq/finlynq/commit/a1e1b51)).** End-to-end inbound email flow replacing the vaporware `@pf.app` addresses. Users forward bank statements to `import-<uuid>@finlynq.com`, and attachments land in a review queue at [/import/pending](src/app/(app)/import/pending/page.tsx) where per-row checkboxes let them approve selectively; approved rows materialize into encrypted `transactions` under the session DEK. Non-import addresses go through a 3-way router: reserved prefixes (`info`/`admin`/`support`/`hello`/`contact`/`sales`/`help`) and named-human matches land in the **admin inbox** at [/admin/inbox](src/app/(app)/admin/inbox/page.tsx) (kept indefinitely, body HTML rendered in a sandboxed iframe); everything else goes to a **trash bin** with a 24-hour TTL and an optional bounce back to the sender (gated on `RESEND_API_KEY` + SPF/DKIM pass — no backscatter). Webhook handler at [/api/import/email-webhook](src/app/api/import/email-webhook/route.ts) accepts Resend JSON (svix-signature-verified via a hand-rolled HMAC-SHA256 verifier at [src/lib/webhooks/svix.ts](src/lib/webhooks/svix.ts) — no new npm dep) with fallbacks: 401 on bad signature, 413 on >10 MB body, 429 on >25 msgs/hour per recipient, 415 on wrong content-type. New schema: `staged_imports`, `staged_transactions`, `incoming_emails` ([scripts/migrate-email-import-staging.sql](scripts/migrate-email-import-staging.sql), applied to prod). The staging design deliberately removes the need for a persistent `email_webhook_dek` envelope — rows sit plaintext for a bounded 14-day window and get encrypted only at user-approve time, shrinking the attack surface. Cleanup cron sweeps both staged imports (14d) and trash (24h) every 30 min. Self-hosted multipart path preserved for backward compat. Full integration plan: [Research/email-import-resend-plan.md](../Research/email-import-resend-plan.md). Ops setup (DNS MX, Resend dashboard, signing secret): [docs/resend-inbound-setup.md](docs/resend-inbound-setup.md). **Current prod state:** code live + DB migrated + placeholder signing secret in env; MX records and real Resend secret still need to be wired per the setup guide.
- **CSV import — column-mapping fallback for non-canonical headers (2026-04-23, commit [df2143c](https://github.com/finlynq/finlynq/commit/df2143c)).** Bank exports (CIBC, RBC, etc.) use headers like `Transaction Date` / `CAD$` / `Description 1` that don't match the canonical `Date`/`Amount`/`Account`/`Payee` parser, so every row was failing as `Invalid date: ""`. [`/api/import/preview`](src/app/api/import/preview/route.ts) now:
  1. Tries the canonical parser first (existing behavior).
  2. **Honors `templateId`** when the user picks a saved template (was silently ignored — latent bug).
  3. Falls back to a **saved template with ≥80% header overlap** via `findBestTemplate()`.
  4. Returns `type: "csv-needs-mapping"` with `headers`, `sampleRows`, and an auto-detected `suggestedMapping` when nothing else works.
  A new [ColumnMappingDialog](src/app/(app)/import/components/column-mapping-dialog.tsx) opens for case 4: user picks columns + a default account (CIBC-style CSVs have no usable account column), sees a live 3-row sample, confirms, and the client (a) parses via `/api/import/csv-map` and (b) **auto-saves the mapping as a template** so subsequent uploads hit case 3 silently. Also tightened `autoDetectColumnMapping()` in [src/lib/import-templates.ts](src/lib/import-templates.ts): currency-suffix matcher for amount (`CAD$` / `USD$` / `$`), exact-match-first pass before substring fallback, and account binds only on literal `account` / `account name` — so `Account Type` (= "MasterCard") doesn't get bound to a user account name.

### Fixed
- **Deleting a transaction that had splits returned 500 (2026-04-23).** `transaction_splits.transaction_id` was defined as a FK to `transactions.id` without `ON DELETE CASCADE`, so every delete path — web `DELETE /api/transactions`, `POST /api/transactions/bulk {action:"delete"}`, MCP `delete_transaction`, MCP `execute_bulk_delete` — failed with a Postgres FK violation whenever the target had any split rows. `wipe-account` was unaffected because it already deletes splits explicitly. Fix: added `{ onDelete: "cascade" }` on the splits FK in [src/db/schema-pg.ts](src/db/schema-pg.ts) and shipped an idempotent migration at [scripts/migrate-tx-splits-cascade.sql](scripts/migrate-tx-splits-cascade.sql) that drops + recreates the constraint with `CASCADE`. The migration introspects `pg_constraint` by (table, column) rather than hardcoding the constraint name, because prod + staging carry the Postgres-default name `transaction_splits_transaction_id_fkey` while dev carries drizzle's name `transaction_splits_transaction_id_transactions_id_fk`. Applied to prod + staging + dev; verified `confdeltype='c'` in all three. No code deploy or restart required — constraint-only change.

  - **Stream C — OAuth hygiene.** [src/lib/oauth.ts](src/lib/oauth.ts): `consumeAuthCode` switched from `SELECT → UPDATE SET used=1` to `DELETE ... Schema migration [scripts/migrate-oauth-revoked-at.sql](scripts/migrate-oauth-revoked-at.sql) adds the column + a partial index `(token) WHERE revoked_at IS NULL` so validation stays fast as revoked rows accumulate. Idempotent (`ADD COLUMN IF NOT EXISTS`). Applied to prod + staging + dev before the code deploy.

- **Re-audit follow-up: API keys now hashed at rest (2026-04-23, commit [b60a439](https://github.com/finlynq/finlynq/commit/b60a439)).** `settings.value` for `key='api_key'` stored the raw `pf_*` secret; a DB read compromised every active key. Switched to `sha256:<64 hex>` at rest.
