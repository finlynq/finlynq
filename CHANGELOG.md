### Changed
- **Restore Category column + expand /transactions column customization (2026-04-29, closes [#20](https://github.com/finlynq/finlynq/issues/20)).** Two related fixes to the `/transactions` table — (a) the Category badge has been silently empty for every Stream-D Phase-3 user since cutover, and the Reports tabs collapsed every category into one "Uncategorized" bucket for the same reason; (b) the column-prefs UX only had a single `localStorage`-backed Portfolio toggle. **Backend — restore decrypted display names through the JOIN.** [`getTransactions()`](src/lib/queries.ts) now selects `categories.name_ct`, `accounts.name_ct`, `accounts.alias_ct`, `portfolio_holdings.symbol`, and `portfolio_holdings.symbol_ct`. `/api/transactions` GET ([route.ts](src/app/api/transactions/route.ts)) and `/api/transactions/linked` ([linked/route.ts](src/app/api/transactions/linked/route.ts)) decrypt those fields with `decryptName()` (auth-tag failure ladder) and strip the `*_ct` columns before serializing. Without this, every Phase-3-NULL'd row returned `categoryName: null` and the Category cell only showed `<Link2 />` / split badges. **Backend — Reports group on stable id.** [/api/reports](src/app/api/reports/route.ts) (income-statement + tax-summary), [/api/reports/trends](src/app/api/reports/trends/route.ts), and [/api/reports/yoy](src/app/api/reports/yoy/route.ts) now `GROUP BY categories.id` (previously the now-NULL `categories.name`) and resolve `name_ct` per group after the SQL aggregation. Behavior change worth flagging: rows with duplicate category names that previously collided now report separately. Balance-sheet's account name decrypts via the same ladder. **New endpoint `/api/settings/tx-columns`** ([route.ts](src/app/api/settings/tx-columns/route.ts)) — GET/PUT round-trips a JSON blob through the existing `settings(key, user_id)` table under `key='tx_table_columns'`. Zod-validated; defaults merged on read so adding a new column id later doesn't force a migration. No schema change required. **Frontend** ([transactions/page.tsx](src/app/%28app%29/transactions/page.tsx)) — replaced the `TxColPrefs = { portfolio: boolean }` localStorage hook with a `ColumnPref[]` (visibility + order) backed by the new endpoint. The Columns dropdown lists every toggleable column + a "Reset to default" action. Native HTML5 `draggable` on `<TableHead>` reorders columns; order persists via the same hook (400ms debounce). Legacy `localStorage["pf-tx-cols-v1"]` Portfolio toggle migrates on first load and is then cleared. The page renders columns from a single `columnPrefs.filter(visible).map(...)` pass instead of the previous hard-coded `TableHead` / `TableCell` blocks. **Out of scope.** `accountDescription` toggle dropped — the column doesn't exist on `accounts` and a schema add is out of bounds for a UI/docs-scoped issue. Drag UX is mouse-only; the checkbox toggles in the dropdown remain a keyboard-accessible alternative for visibility (order remains drag-only).
- **`transactions.portfolio_holding` retired end-to-end (Phase 5 + 6, 2026-04-29, [#18](https://github.com/finlynq/finlynq/pull/18)).** The legacy encrypted text column on `transactions` is gone; the integer FK `transactions.portfolio_holding_id → portfolio_holdings.id` (introduced 2026-04-26 in [#7](https://github.com/finlynq/finlynq/pull/7)) is now the sole source of truth, and the holding name is sourced via JOIN to `portfolio_holdings`. **Gate fix (root cause).** The `withoutFk` count on prod was permanently stuck at 4 — all 4 rows belonged to the demo user. [scripts/seed-demo.ts](scripts/seed-demo.ts) inserted investment transactions BEFORE the matching `portfolio_holdings` rows, so the seed never set the FK. The lazy-on-login backfill caught them when the demo user logged in, but the nightly reseed wiped the data and re-inserted 4 broken rows, so `withoutFk ≥ 4` between reset and next demo login. Fix: insert holdings first, capture their ids, bind `portfolio_holding_id` directly. After the seed-script fix + a manual reseed, gate dropped to 0. **Phase 5 — code cutover ([a80dcbb](https://github.com/finlynq/finlynq/commit/a80dcbb)):** migration [scripts/migrate-tx-portfolio-holding-phase5-null.sql](scripts/migrate-tx-portfolio-holding-phase5-null.sql) NULLed the legacy column on every backfilled row (1200 rows on prod). Dropped `"portfolioHolding"` from `TX_ENCRYPTED_FIELDS`. Deleted orphan-fallback decrypt loops in `/api/portfolio/overview`, `holdings-value.ts`, and `mcp-server` `aggregateHoldings()` — the portfolio aggregator no longer needs the DEK. POST/PUT `/api/transactions` `delete data.portfolioHolding` before persisting; GET resolves `portfolioHolding` (name) via the Stream D `name_lookup` HMAC and filters SQL-side. **Phase 6 — DROP COLUMN ([d28258e](https://github.com/finlynq/finlynq/commit/d28258e)):** migration [scripts/migrate-tx-portfolio-holding-phase6-drop-column.sql](scripts/migrate-tx-portfolio-holding-phase6-drop-column.sql) with a defensive pre-check that refuses to run if any non-NULL plaintext remains. `getTransactions` JOINs `portfolio_holdings` and surfaces `{ portfolioHoldingId, portfolioHoldingName, portfolioHoldingNameCt }`; the `/api/transactions` GET handler decrypts `name_ct` (with plaintext name fallback) and presents it to the UI as `portfolioHolding` so the client contract is unchanged. Schema-pg.ts removes the column from the Drizzle definition. Import pipeline stops encrypting the column; the in-memory `portfolioHolding` field stays on the insertable shape (resolver uses it) and is stripped before `db.insert` via destructuring. **Retired (deleted):** `src/lib/crypto/portfolio-holding-fk-backfill.ts`, `src/app/api/admin/portfolio-holding-fk-progress/route.ts`. Removed the `enqueuePortfolioHoldingFkBackfill` call sites in `auth/login` + `auth/mfa/verify`. **Follow-up fix (post-merge).** Caught after the merge: `analyze_holding` and `get_investment_insights` in both HTTP MCP ([mcp-server/register-tools-pg.ts](mcp-server/register-tools-pg.ts)) and stdio MCP ([mcp-server/register-core-tools.ts](mcp-server/register-core-tools.ts)) still SELECTed and grouped by the dropped column. They were not on the original Phase 5 hit-list (only the three "orphan-fallback" sites were called out). Switched all of them to FK + JOIN aggregation. Without this fix, every MCP `analyze_holding` / `get_investment_insights` invocation on prod would have raised `column "portfolio_holding" does not exist`. **Order of operations.** Phase 5 was migration-first then code (NULLing is backwards-compatible — old code reads NULL gracefully). Phase 6 was code-first then migration (DROP COLUMN is backwards-incompatible — old code SELECTs the column). [PHASE5_PORTFOLIO_HOLDING_FK.md](../PHASE5_PORTFOLIO_HOLDING_FK.md) drove the work; marked complete.
- **Stream D Phase 3: per-user lazy plaintext NULL on login (2026-04-29, closes [#7](https://github.com/finlynq/finlynq/issues/7)).** Replaces the eager all-or-nothing [`scripts/migrate-stream-d-phase3-null.sql`](scripts/migrate-stream-d-phase3-null.sql) (applied to prod 2026-04-24, blocked on staging + dev because demo's lazy backfill never finished and the strict pre-check rolled back the whole transaction) with a per-user lazy variant that runs on each successful login. Three gates before NULLing a user's plaintext name columns: (1) `users.plaintext_nulled_at IS NULL`, (2) the user's own Stream D backfill is complete (no plaintext-but-no-ct rows across the six Stream D tables), (3) a sample row's `name_ct` actually decrypts with the cached DEK. Stragglers and DEK-mismatch users keep their dual-write fallback indefinitely; healthy users get Phase 3'd one-by-one. Doubles as the diagnostic for the unresolved pathfinder DEK-mismatch — the `[phase3-null] user=… table=… sample decrypt failed` warn names the user, the table, and the underlying error, replacing the planned `PF_CRYPTO_DEBUG=1` fingerprint deploy. New helper [src/lib/crypto/stream-d-phase3-null.ts](src/lib/crypto/stream-d-phase3-null.ts) exposes `nullPlaintextIfReady(userId, dek)` (atomic single-transaction NULL across the six tables, sets `users.plaintext_nulled_at`) and `enqueuePhase3NullIfReady` (fire-and-forget login wrapper that never blocks login). Wired one line each into [src/app/api/auth/login/route.ts](src/app/api/auth/login/route.ts) + [src/app/api/auth/mfa/verify/route.ts](src/app/api/auth/mfa/verify/route.ts) right after the existing `enqueueStreamDBackfill` call. Schema migration [scripts/migrate-stream-d-phase3-per-user.sql](scripts/migrate-stream-d-phase3-per-user.sql) drops NOT NULL on the six plaintext name columns (idempotent — prod's eager migration already did the same) and adds `users.plaintext_nulled_at text`. **Apply per env BEFORE deploying the matching code** ([docs/migrations.md](docs/migrations.md) playbook). Applied to dev on 2026-04-29; required on staging + prod before this code ships. Drizzle schema intentionally unchanged — helper writes via raw SQL bypassing Drizzle's typing so the six plaintext name columns can stay `.notNull()` and downstream consumers don't have to handle `string | null` cascades. Admin status `GET /api/admin/phase3-null-progress` reports `{ usersTotal, usersDone, usersPending, pendingDetail[] }` with per-user blocking-row counts. Eager migration retired with a SUPERSEDED comment header — kept as the record of prod's 2026-04-24 eager run.
- **`deploy.sh` no longer runs `npm run db:push` (2026-04-29, closes [#5](https://github.com/finlynq/finlynq/issues/5)).** The step looked like automated schema sync but silently no-op'd on every deploy: `DATABASE_URL` from the systemd unit doesn't survive the `sudo -u $REPO_OWNER` hop in `run_as`, so Drizzle-Kit fell back to its hardcoded `postgres://localhost:5432/pf` default and either failed authentication silently or connected to the wrong DB. Symptom surfaced shipping the investment-account constraint — Deploy Dev reported success, but `accounts.is_investment` wasn't actually added to the dev DB and `/api/dashboard` 500'd until the SQL migration was run by hand. Removed the step in favor of the existing per-env psql playbook in [docs/migrations.md](docs/migrations.md), which is what we already use in practice. Deploy is now strictly "pull + install + build + restart"; the database is the human's responsibility before pushing the matching code. CLAUDE.md gotcha + DEPLOY.md updated to match.

Also added a "Columns" affordance in the filter bar — a `DropdownMenu` with a "Portfolio" checkbox, persisted to `localStorage` key `pf-tx-cols-v1` and **off by default** so users without investment activity don't see a column of dashes. ### Added
- **Investment-account constraint — every transaction in a flagged account must reference a portfolio holding (2026-04-28).** New `accounts.is_investment` boolean flag with a per-account opt-in checkbox in the account edit dialog. Once set, every transaction in that account must reference a `portfolio_holdings` row (FK `transactions.portfolio_holding_id`) — trades point at their security; cash legs (deposits, dividends paid as cash, fees, transfers) point at a per-account "Cash" holding using the existing currency-as-holding pattern. Closes the long-standing gap where the portfolio aggregator's orphan-fallback path silently dropped null-FK / null-text rows from cost-basis math, dividend totals, and per-holding performance — even though those rows still affected the cash balance.

  - **Schema** — `ALTER TABLE accounts ADD COLUMN is_investment boolean NOT NULL DEFAULT false` ([scripts/migrate-accounts-is-investment.sql](scripts/migrate-accounts-is-investment.sql)). Idempotent. No DB CHECK constraint — enforcement is application-layer (mirrors the four-check transfer-pair pattern in [src/lib/transfer.ts](src/lib/transfer.ts) since PostgreSQL can't express "NOT NULL only when accounts.is_investment is true" without a trigger). The migration also runs a heuristic backfill (any account with ≥1 `portfolio_holdings` row pre-migration is flagged), ensures a per-account `name='Cash', symbol=NULL, currency=accounts.currency` holding for each flagged account, and points all unattributed null-FK / null-text rows at it. Rows with the legacy plaintext `portfolio_holding` text column populated stay untouched — Phase-4 lazy resolver routes them to their actual holding on next login.

  - **Centralized helper** [src/lib/investment-account.ts](src/lib/investment-account.ts) — `requireHoldingForInvestmentAccount` (strict, throws `InvestmentHoldingRequiredError` for REST + MCP HTTP write tools), `defaultHoldingForInvestmentAccount` (permissive — returns the per-account Cash holding id for import + transfer-pair construction), `getOrCreateCashHolding` (find-or-create using the same plaintext+lookup dual-index pattern as the existing portfolio-holding-resolver, tolerates `dek=null` for stdio MCP), and `backfillInvestmentAccount` (the migration's logic scoped to a single account, called from `PUT /api/accounts` on a false→true toggle).

  - **Call-site coverage** — `createTransaction` / `updateTransaction` ([src/lib/queries.ts](src/lib/queries.ts)), `createTransferPair` + `createTransferPairViaSql` ([src/lib/transfer.ts](src/lib/transfer.ts), pure-cash legs default to Cash), `executeImport` ([src/lib/import-pipeline.ts](src/lib/import-pipeline.ts), post-resolver Cash-default pass per investment account), MCP HTTP `record_transaction` / `bulk_record_transactions` / `update_transaction` ([mcp-server/register-tools-pg.ts](mcp-server/register-tools-pg.ts)) take the strict path. The orphan-aggregator path stays in place — staging/dev users still rely on it until Phase 5 cutover.

  - **Stdio MCP** `record_transaction` / `bulk_record_transactions` refuse writes to investment accounts with a pointer to the HTTP MCP at `/mcp` — stdio MCP doesn't expose `portfolioHolding` parameters, so it can't satisfy the constraint. Documented as a self-hosted limitation alongside the existing "stdio writes are plaintext" gotcha. Stdio `record_transfer` is unaffected because it routes through `createTransferPairViaSql` which got the Cash-default treatment.

  - **Account toggle side-effect** — `PUT /api/accounts` with `isInvestment: true` runs the same backfill the SQL migration does (Cash holding ensure + null-FK reassignment) atomically with the flag flip. Backfill failure surfaces a 500 with an actionable error so the user can retry. `true → false` just flips the flag; existing data stays attributed.

  - **Admin queue** `GET /api/admin/investment-orphans` ([src/app/api/admin/investment-orphans/route.ts](src/app/api/admin/investment-orphans/route.ts)) reports rows in investment accounts that still have NULL FK after Phase-4 lazy backfill — typically legacy WP imports where the connector failed to populate even the plaintext text column. Surfaces a count + capped 200-row sample, no payee plaintext through this endpoint.

  - **UI** "Investment account" checkbox added to the account edit dialog at [src/app/(app)/accounts/page.tsx](src/app/%28app%29/accounts/page.tsx); `AccountBalance` + `getAccountBalances` plumb `isInvestment` through.

  - **Migration playbook** in [docs/migrations.md](docs/migrations.md) under `accounts-is-investment`. **Run the SQL BEFORE deploying the matching code** so the application-layer constraint check has the column to read.

- **MCP holding-id ergonomics — bind transactions to positions by name OR ticker (2026-04-28, commits [`ca0a117`](https://github.com/finlynq/finlynq/commit/ca0a117) + [`f429c6f`](https://github.com/finlynq/finlynq/commit/f429c6f)).** Two follow-ups closing the gap reported in the [2026-04-28 Huron Sale conversation](https://claude.ai/share/a066166f-c5ee-4abe-a01c-25868afafb1c) — Claude recorded a stock sale via the MCP, the `quantity` field made it through (the [`7325448`](https://github.com/finlynq/finlynq/commit/7325448) fix from earlier the same day), but `transactions.portfolio_holding_id` was left NULL so the position never moved. The Edit Transaction dialog rendered `Portfolio Holding: None` for a row with `quantity=-48`. Root cause was upstream of the write tools: `get_portfolio_analysis` and `analyze_holding` never returned the holding's integer id, so an agent had no way to populate the existing `portfolioHoldingId` parameter except by calling `add_portfolio_holding` (which would create a duplicate). The aggregator's `accumulate()` already plumbed `holding_id` through ("first non-null id wins"); the response mappers were dropping it on the floor. Same shape on the stdio twins.

  - **[`ca0a117`](https://github.com/finlynq/finlynq/commit/ca0a117) — Phase 1 (id surfacing) + Phase 2 (name-string write ergonomics).** `get_portfolio_analysis` now emits `id: <number>` per holding; `analyze_holding` emits `holdingId: <number>` at the top level. Both transports. `analyze_holding`'s HTTP find is tightened to prefer rows whose decrypted `portfolio_holding === holdingName` so payee-only fuzzy matches don't surface another holding's id when the analyzed string was a non-investment cash payee like `"Huron Sale"`. The three HTTP write tools (`record_transaction` / `bulk_record_transactions` / `update_transaction`) accept a new `portfolioHolding: string` parameter alongside the integer `portfolioHoldingId`. New helper `resolvePortfolioHoldingByName` in [register-tools-pg.ts](mcp-server/register-tools-pg.ts) — single-shot SELECT, exact case-insensitive match scoped to the resolved account, dual-cohort handling (plaintext `name` + `name_lookup` HMAC for Phase-3 NULL'd rows). Lookup-only — preserves the existing policy that MCP write tools never auto-create holdings (only the import pipeline does). Errors with a candidate list on miss; errors when both `portfolioHolding` and `portfolioHoldingId` are passed and resolve to different ids ("I named X but you bound Y" is worse than rejecting). Stdio write tools unchanged — they still don't bind `portfolio_holding` per the existing carve-out.

  - **[`f429c6f`](https://github.com/finlynq/finlynq/commit/f429c6f) — Phase 2.5: ticker resolution.** Follow-up after verifying Phase 2 on dev exposed an inconsistency: `update_transaction(portfolioHolding: "HURN")` errored "not found" while `"Huron Consulting Group Inc."` worked, even though `get_portfolio_analysis(symbols: ["HURN"])` already matched the ticker via the `symbol` column. Resolver SQL extended to also check `LOWER(symbol)` plaintext + `symbol_lookup` HMAC alongside name/name_lookup — the same `nameLookup(dek, trimmed)` HMAC value matches both columns since the HMAC is over trimmed-lowercase input regardless of source. Candidate-list error reformatted as `"Name (TICKER)"` so the agent sees both identifiers. `analyze_holding(symbol)` HTTP path now JOINs `ph.symbol_ct`, decrypts per-row, and the in-memory filter checks symbol via **exact** case-insensitive equality (substring on a 3-4 char ticker would match "GE" inside "ORANGE" — name + payee retain substring matching for the long-string ergonomics). Stdio twin same shape — `LEFT JOIN portfolio_holdings ph` + `OR LOWER(ph.symbol) = LOWER(?)` in the WHERE. The tool's description always claimed "fuzzy match on name OR symbol"; this commit makes it honest.

  - **End-to-end verified on dev** ([Deploy Dev runs `25076508331` + `25077631853`](https://github.com/finlynq/finlynq/actions)): (a) `get_portfolio_analysis(symbols: ["Huron"])` → response includes `id: 452`. (f) `analyze_holding(symbol: "HURN")` → finds Huron, returns `holdingId: 452`, full transaction history (was: "No transactions found").

- **First-class Transfer flow + in-kind / portfolio support (2026-04-28, branch [`feat/transfer-flow`](https://github.com/finlynq/finlynq/tree/feat/transfer-flow) merged via [`5d945a6`](https://github.com/finlynq/finlynq/commit/5d945a6); five commits cf411ba → a81d9a7 → c57fe1b → 2c320fc → 269b381 → c533781).** End-to-end primitive for "move money/shares between two of my accounts." - **Cross-tenant FK leak in backup-restore + atomic, strict wipe (2026-04-27).** Two commits ([`5571070`](https://github.com/finlynq/finlynq/commit/5571070) → [`ad87419`](https://github.com/finlynq/finlynq/commit/ad87419)) closing one root cause and one wipe-side hardening, surfaced by the admin's "Wipe Account" 500ing with `Failed query: delete from "accounts" where "accounts"."user_id" = $1`.

  **Root cause — `/api/data/import` strip().** The JSON backup-restore handler at [src/app/api/data/import/route.ts](src/app/api/data/import/route.ts) used a generic `strip(rows, userId)` helper that only deleted `id` and forced `userId`, leaving `accountId` / `categoryId` / `assignCategoryId` columns UNCHANGED from the source backup. The route already builds an `accountIdMap` and `categoryIdMap` (old-id → new-id, populated when accounts and categories are inserted), and `transactions` + `transaction_splits` had explicit per-table remap blocks — but the strip-based inserts for `portfolio_holdings`, `loans`, `goals`, `snapshots`, `recurring_transactions`, `subscriptions`, `budgets`, `budget_templates`, and `transaction_rules` didn't touch the maps. So restoring user A's backup as user B silently wrote rows owned by B but with FK references to A's account/category integer IDs — accepted by Postgres because those IDs still existed in the same DB, producing a cross-tenant data leak that also became a wipe-blocker via FK 23503 the next time A tried to delete their account. Fix: `strip()` now takes optional `{ accountIdMap, categoryIdMap }`, remaps the FK columns in-place, and **throws on an unmapped FK** rather than passing the raw id through (fail-loud — a corrupt or partial backup is now a 500 instead of a silent leak). Every restore call site that touches an FK-bearing table passes the right map; tables with no FKs keep the old single-arg call. Concrete prod blast: a test user `pathfinder` restored the admin's backup, which created 60 portfolio_holdings owned by `pathfinder` but pointing at admin's accounts (IDs 437–461). All 60 rows were cleaned up via SQL as part of the same fix — they had no business existing.

  **Wipe-side hardening — atomicity + strict isolation.** [`wipeUserDataAndRewrap`](src/lib/auth/queries.ts) used to run each `db.delete(...)` as its own auto-commit, so a late failure (like the FK violation above) left the user signed in to a half-wiped account whose DEK was never rotated, and re-running the wipe just looped on the same FK. Wrap the entire delete sequence + the final DEK rewrap in a single `db.transaction(async (tx) => ...)` — atomic. Every delete still filters strictly by `user_id` only — **never by FK reach** — so the wipe can ONLY remove rows owned by the wiping user. If a cross-tenant row exists, the wipe fails with FK 23503 + transaction rollback rather than silently destroying another user's data. (An earlier version of `5571070` did add a defensive `account_id`-keyed sweep across all 8 referrer tables; that was reverted in `ad87419` because the right fix is to prevent cross-tenant rows from being created in the first place — the `strip()` change above.) The `mcp_uploads` file unlink loop runs BEFORE the DB transaction so we'd rather leak a DB row than orphan a plaintext file on disk if the wipe later fails.

- **Decryption resilience for stale-DEK ciphertext (2026-04-27).** Three-commit batch ([`152e4e6`](https://github.com/finlynq/finlynq/commit/152e4e6) → [`b45c9cf`](https://github.com/finlynq/finlynq/commit/b45c9cf) → [`efd7de2`](https://github.com/finlynq/finlynq/commit/efd7de2)) that hardens every read path against AES-GCM auth-tag failures so a single undecryptable row no longer 500s the whole endpoint. On failure they log a single `[envelope] decryptName failed` warn (so the underlying mismatch stays observable) and return the plaintext column when present, or the raw `v1:...` ciphertext as a UI marker when there's no fallback. Encryption is defense-in-depth — when the dual-write plaintext is still on disk, rendering it beats crashing the dashboard.

  **`tryDecryptField` helper** added in [envelope.ts](src/lib/crypto/envelope.ts). Same as `decryptField` but returns `null` (not the raw ciphertext, not a throw) on auth-tag failure, so callers can use the standard `tryDecryptField(...) ?? plaintextFallback` pattern. The two-commit chain is instructive: v1 of the helper returned the raw ciphertext on failure, which silently broke `?? plaintext` fallbacks at every call site (`"v1:..."` is truthy, so `??` skipped the fallback). Server extracts via `jszip` (no native deps), parses via a hand-rolled RFC-4180 CSV reader ([csv.ts](packages/import-connectors/src/wealthposition/csv.ts)), and transforms into Finlynq's `RawTransaction[]` + `transaction_splits[]` + `portfolio_holdings[]`. Routes: [`/api/import/connectors/wealthposition/zip-{probe,preview,execute}`](src/app/api/import/connectors/wealthposition/). Instant (no rate limits) and handles 100% of transaction shapes. Portfolio.csv is the key: it's the authoritative holding → brokerage + ticker symbol mapping, which WP's API never exposes. Verified against a real 6,596-row export → 5,817 flat txs + 111 split parents + 60 portfolio_holdings rows + 0 transform errors.

  **Secondary: API key.** Live [WP API](https://www.wealthposition.com/api/v0.1) access, kept for balance reconciliation against `/account_balances`. Key is AES-GCM-encrypted under the user's DEK and persisted in `settings` under `connector:wealthposition:credentials`. Rate-limited client (1 req/s) with SHA-keyed global bucket on `globalThis` so state survives across HTTP requests in the same process; automatic retry with jittered backoff on `RATE_LIMIT_ERROR` / HTTP 429. Original plan had the API as the primary import path — pivoted to ZIP after discovering the API's `/accounts` flattens every position (Bitcoin, Ethereum, every stock ticker) as a separate top-level account with no way to tie them back to their brokerage.

  **Transform rules (ZIP path).** The CSV uses a `#SPLIT#` sentinel to preserve multi-leg transactions (parent row has `account=<real>` + `categorization=#SPLIT#`, consecutive children have `account=#SPLIT#`). Walker groups parent + consecutive children, classifies each child as category / account / holding, and emits:
    - **Simple 1A + 1C** → 1 Finlynq flat tx (4,204 rows, 63.5%).
    - **Cash + #SPLIT# + N categories** → 1 parent + N rows in `transaction_splits` (paycheck splits).
    - **Cash + #SPLIT# + N holdings** → parent cash leg preserved on the funding account + N position legs routed via `Portfolio.csv` to each holding's own brokerage (so a stock buy funded from RBC lands the cash leg on RBC and the shares on Questrade - RESP, each with `portfolioHolding = holding_name` + `quantity`).
    - **Cash + #SPLIT# + N accounts** → transfer set (parent + N flat txs, all tagged with the Transfer category).
    - **Orphan #SPLIT# rows** → surfaced as transform errors (shown in the preview dialog, not dropped silently).

  **Mapping + reconciliation.** Probe returns Finlynq's live accounts + categories so the mapping dialog can render names (not opaque ids); stale overrides (pointing at accounts deleted outside the import flow) are filtered client-side before opening the dialog and re-validated server-side in `materializeZipMapping`. Reconciliation endpoint fetches WP's `/accounts` once to build a `name→UUID` bridge, so CSV synthetic ids (`csv:acct:<name>`) match WP's real UUIDs from `/account_balances`. After a successful import, reconciliation runs inline when an API key is on file — per-account WP-vs-Finlynq diff table surfaces in the summary card, with an "Add opening balance" affordance for constant-offset mismatches.
