### Added
- **Email import via Resend Inbound with staging review queue (2026-04-23, commits [8dbf0db](https://github.com/finlynq/finlynq/commit/8dbf0db) → [a1e1b51](https://github.com/finlynq/finlynq/commit/a1e1b51)).** End-to-end inbound email flow replacing the vaporware `@pf.app` addresses. Users forward bank statements to `import-<uuid>@finlynq.com`, and attachments land in a review queue at [/import/pending](src/app/(app)/import/pending/page.tsx) where per-row checkboxes let them approve selectively; approved rows materialize into encrypted `transactions` under the session DEK. Non-import addresses go through a 3-way router: reserved prefixes (`info`/`admin`/`support`/`hello`/`contact`/`sales`/`help`) and named-human matches land in the **admin inbox** at [/admin/inbox](src/app/(app)/admin/inbox/page.tsx) (kept indefinitely, body HTML rendered in a sandboxed iframe); everything else goes to a **trash bin** with a 24-hour TTL and an optional bounce back to the sender (gated on `RESEND_API_KEY` + SPF/DKIM pass — no backscatter). Webhook handler at [/api/import/email-webhook](src/app/api/import/email-webhook/route.ts) accepts Resend JSON (svix-signature-verified via a hand-rolled HMAC-SHA256 verifier at [src/lib/webhooks/svix.ts](src/lib/webhooks/svix.ts) — no new npm dep) with fallbacks: 401 on bad signature, 413 on >10 MB body, 429 on >25 msgs/hour per recipient, 415 on wrong content-type. New schema: `staged_imports`, `staged_transactions`, `incoming_emails` ([scripts/migrate-email-import-staging.sql](scripts/migrate-email-import-staging.sql), applied to prod). The staging design deliberately removes the need for a persistent `email_webhook_dek` envelope — rows sit plaintext for a bounded 14-day window and get encrypted only at user-approve time, shrinking the attack surface. Cleanup cron sweeps both staged imports (14d) and trash (24h) every 30 min. Self-hosted multipart path preserved for backward compat. Full integration plan: [Research/email-import-resend-plan.md](../Research/email-import-resend-plan.md). Ops setup (DNS MX, Resend dashboard, signing secret): [docs/resend-inbound-setup.md](docs/resend-inbound-setup.md). **Current prod state:** code live + DB migrated + placeholder signing secret in env; MX records and real Resend secret still need to be wired per the setup guide.
- **CSV import — column-mapping fallback for non-canonical headers (2026-04-23, commit [df2143c](https://github.com/finlynq/finlynq/commit/df2143c)).** Bank exports (CIBC, RBC, etc.) use headers like `Transaction Date` / `CAD$` / `Description 1` that don't match the canonical `Date`/`Amount`/`Account`/`Payee` parser, so every row was failing as `Invalid date: ""`. [`/api/import/preview`](src/app/api/import/preview/route.ts) now:
  1. Tries the canonical parser first (existing behavior).
  2. **Honors `templateId`** when the user picks a saved template (was silently ignored — latent bug).
  3. Falls back to a **saved template with ≥80% header overlap** via `findBestTemplate()`.
  4. Returns `type: "csv-needs-mapping"` with `headers`, `sampleRows`, and an auto-detected `suggestedMapping` when nothing else works.
  A new [ColumnMappingDialog](src/app/(app)/import/components/column-mapping-dialog.tsx) opens for case 4: user picks columns + a default account (CIBC-style CSVs have no usable account column), sees a live 3-row sample, confirms, and the client (a) parses via `/api/import/csv-map` and (b) **auto-saves the mapping as a template** so subsequent uploads hit case 3 silently. Also tightened `autoDetectColumnMapping()` in [src/lib/import-templates.ts](src/lib/import-templates.ts): currency-suffix matcher for amount (`CAD$` / `USD$` / `$`), exact-match-first pass before substring fallback, and account binds only on literal `account` / `account name` — so `Account Type` (= "MasterCard") doesn't get bound to a user account name.

### Security
- **V2 remediation batch (2026-04-23, commits [0479d88](https://github.com/finlynq/finlynq/commit/0479d88) + [5948c91](https://github.com/finlynq/finlynq/commit/5948c91) + [689c199](https://github.com/finlynq/finlynq/commit/689c199)).** Four of the five streams from [AUDIT_REMEDIATION_PLAN_V2.md](../AUDIT_REMEDIATION_PLAN_V2.md) shipped in one push. Only Stream D (encryption coverage for `accounts.name` et al.) remains — explicitly scoped as multi-session.
  - **Stream A — `xlsx@0.18.5` CVE-2023-30533.** Swapped to `exceljs@^4.4.0`. [src/lib/excel-parser.ts](src/lib/excel-parser.ts) rewritten against the new API; both exports (`parseExcelSheets`, `extractExcelRows`) are now async because `workbook.xlsx.load()` is Promise-based. Added a `cellToString()` helper normalizing exceljs's richer cell shapes (rich text, hyperlink, formula `{result}`, Date, error) to the plain strings the downstream row logic expects. Four callers (`/api/import/preview`, `/api/import/excel-map`, `/api/import/email-webhook`, `src/lib/email-import/parse-attachments`) updated to `await`. `load()` cast through `any` because exceljs's `index.d.ts` declares its own `Buffer extends ArrayBuffer` that doesn't line up with Node's `Buffer<ArrayBufferLike>` under modern `@types/node` — runtime is fine. Package-lock regenerated on the Linux deploy host (same c08b194 pattern) to preserve `@emnapi/runtime` + `@esbuild/linux-x64` optional deps that Windows npm strips.
  - **Stream B — CSP flip Report-Only → enforced.** One-line header-name change in [next.config.ts](next.config.ts). Watch window in 50a1742 was clean. Verified on prod: `curl -I https://finlynq.com/` returns `Content-Security-Policy:` (no `-Report-Only` suffix) on both public (`/`, with GA whitelist) and app (`/dashboard`, locked-down) routes — the route-aware middleware CSP takes precedence on website paths and the `next.config.ts` CSP covers the rest. Rollback is a one-line flip of the header name back.
  - **Stream C — OAuth hygiene.** [src/lib/oauth.ts](src/lib/oauth.ts): `consumeAuthCode` switched from `SELECT → UPDATE SET used=1` to `DELETE ... RETURNING` for atomic claim (concurrent exchanges on the same code can't both succeed; spent codes + their `dek_wrapped` are gone from DB immediately). `refreshAccessToken` now does rotation + reuse detection — rows carry a `revoked_at timestamptz`, refresh flips the live row to revoked atomically (`UPDATE ... WHERE revoked_at IS NULL RETURNING`), and a subsequent presentation of the same (already-revoked) refresh token triggers revocation of every live access token for that user (token-theft containment). `validateOauthToken` filters `revoked_at IS NULL`. Schema migration [scripts/migrate-oauth-revoked-at.sql](scripts/migrate-oauth-revoked-at.sql) adds the column + a partial index `(token) WHERE revoked_at IS NULL` so validation stays fast as revoked rows accumulate. Idempotent (`ADD COLUMN IF NOT EXISTS`). Applied to prod + staging + dev before the code deploy.
  - **Stream E — `mcp_uploads`.** Verified on prod 2026-04-23: table was already present with full column set, both expected indexes, and `users(id)` FK. CLAUDE.md's stale bullet got cleared with no code change.
  - **Deploy:** one GHA run ([24851507625](https://github.com/finlynq/finlynq/actions/runs/24851507625)), typecheck + build + SSH deploy green. No downtime. Schema migration for Stream C applied via `psql` before the code push.
- **Re-audit follow-up: API keys now hashed at rest (2026-04-23, commit [b60a439](https://github.com/finlynq/finlynq/commit/b60a439)).** `settings.value` for `key='api_key'` stored the raw `pf_*` secret; a DB read compromised every active key. Switched to `sha256:<64 hex>` at rest. [src/lib/api-auth.ts](src/lib/api-auth.ts) hashes on write (`getOrCreateApiKey`, `regenerateApiKey`) and on validation lookup; a raw-value fallback in `validateApiKey` silently migrates legacy rows on their next use, so there was no deploy-vs-migration ordering window. The DEK-envelope path is untouched — the raw key provided per-request is still the unwrap input; only the server's stored copy changed. New response contract on `/api/settings/api-key` GET: `{ apiKey: string | null, hasKey: true, hasDekWrap }` — raw key is returned only on first creation or regenerate, `null` thereafter. Settings + MCP-guide pages updated with a "save this key now" banner on fresh keys and a "regenerate to view" state when only a hash is on file. One-shot sweep at [scripts/migrate-hash-api-keys.ts](scripts/migrate-hash-api-keys.ts) hashed all dormant keys on prod (5 rows).
- **Re-audit follow-up: `update_goal` SQL injection + 128-bit email-import token (2026-04-23, commit [df1de92](https://github.com/finlynq/finlynq/commit/df1de92)).** Two regressions surfaced during the 2026-04-23 re-audit, both fixed in one commit.
  - `update_goal` in [mcp-server/register-tools-pg.ts](mcp-server/register-tools-pg.ts) was using `sql.raw(updates.join(", "))` with manual quote escaping on `name` and unescaped `deadline`/`status` interpolation — same class of SQLi as the `update_account` bug fixed in `50a1742`, same file, missed in the V1 sweep. Switched to the parameterized `sql.join` pattern used by `update_account` / `update_subscription`, plus a `rowCount` ownership check. Scope of the bug was self-mutation (user id filter still in WHERE) but the class matters.
  - Email-import tokens regressed from 128 bits to 32 bits when the Resend Phase A code (`385cda5`) rewrote `/api/import/email-config` — `randomUUID().slice(0, 8)` instead of the `randomBytes(16).toString("hex")` H6 fix from the April 11 audit. Restored at [src/app/api/import/email-config/route.ts](src/app/api/import/email-config/route.ts) and widened the [address-router.ts](src/lib/email-import/address-router.ts) regex to `[a-f0-9]{8,64}` so any existing 8-hex addresses keep routing during the transition — the DB lookup on the full address is the actual authorization.
- **Plan for remaining exposures at [AUDIT_REMEDIATION_PLAN_V2.md](../AUDIT_REMEDIATION_PLAN_V2.md)**: `xlsx` CVE, CSP flip to enforced, OAuth hygiene, encryption coverage for `accounts.name` et al., and the `mcp_uploads` schema drift on prod.
- **Full audit remediation (2026-04-22).** Addressed 3 critical, 4 high, 4 medium findings in one commit. Plan: [AUDIT_REMEDIATION_PLAN.md](../AUDIT_REMEDIATION_PLAN.md).
  - **Critical — stdio MCP user isolation.** `mcp-server/index.ts` now requires `PF_USER_ID` at boot (exits otherwise). ~68 SQL queries across `register-core-tools.ts`, `tools-v2.ts`, `tools-import-templates.ts` now filter by `user_id`. Every UPDATE/DELETE has an ownership pre-check. INSERTs bind userId from closure, never from tool arguments. Before the fix, a stdio caller against a multi-user DB could read and destructively write across all tenants.
  - **Critical — SQL injection in MCP `update_account`.** Replaced `sql.raw()` + manual quote escaping with parameterized `sql` fragments.
  - **Critical — CSV import cross-user attach.** All name-based lookups in `src/lib/csv-parser.ts` now filter by userId. Before, one user's import could attach transactions to another user's account if names collided.
  - **High — `encryptionV` bump in `wipeUserDataAndRewrap`.** Multi-tab password resets now correctly invalidate cached DEKs.
  - **High — `PF_JWT_SECRET` fatal in prod.** `src/lib/auth/jwt.ts` throws at module load if missing in production; dev still falls back with a one-time warn.
  - **High — OAuth DCR rate limit.** `/api/oauth/register` capped at 10 registrations/hour/IP.
  - **High — Security headers in `next.config.ts`.** HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy enforced. CSP ships as Report-Only — flip to enforced after a week of clean console.
  - **Medium — Input/request caps.** `/api/chat` message ≤ 2000 chars. `/api/data/import` and `/api/import/execute` reject bodies > 20 MB (413) and imports > 50 000 transactions (422).
  - **Medium — Verified OAuth DEK envelope** end-to-end in `src/lib/oauth.ts` (authorize → token-exchange → refresh all wrap/unwrap correctly; audit's "unknown" was a wrong path).
  - **Deferred:** `xlsx@0.18.5` has CVE-2023-30533 — fix lives only in SheetJS paid/CDN channel, not npm. `pdf-parser.ts:55` pre-existing type error.
