### Changed
- **MCP `bulk_update`: structured `unappliedChanges` + `sampleAfter.category` re-hydration (2026-05-01, closes [#93](https://github.com/finlynq/finlynq/issues/93)).** `preview_bulk_update` / `execute_bulk_update` (HTTP + stdio) now surface every requested change that failed to resolve as `{ field: string, requestedValue: unknown, reason: string }` (was `{ key, reason }`). `field` replaces `key`; `requestedValue` carries the value the caller passed (e.g. `"Credit Interesst"` when category lookup failed) so callers don't have to regex the `reason` string to recover what they sent. Identical shape on HTTP ([mcp-server/register-tools-pg.ts](mcp-server/register-tools-pg.ts)) and stdio ([mcp-server/register-core-tools.ts](mcp-server/register-core-tools.ts)); identical on both `preview_*` and `execute_*`. Tool descriptions and the matching `finlynq_help` snippets updated symmetrically. **`sampleAfter.category` re-hydration.** When `changes.category` (name) resolves successfully, the preview's `sampleAfter.category` (the joined display-name string) now reflects the resolved category instead of the row's old category. Previously `sampleAfter.category_id` flipped to the new id but the name didn't change, so a successful update looked like a no-op. The resolver writes the resolved name onto a new preview-only `ResolvedChanges.category_name` field and `applyChangesToRow` reads it back when building `sampleAfter`. `category_name` is NOT a DB column and `commitBulkUpdate` doesn't write it — purely preview metadata. The "all changes failed to resolve" abort guard in `execute_bulk_update` filters `category_name` out of the resolved-key count, so the guard still fires correctly when only an unresolvable name was passed. **Caller-visible breaking change.** Any external client that parsed `unappliedChanges[].key` will need to read `.field`. **No DB schema change. No new write paths.** Confirmation-token derivation (still signs the user-supplied `changes`, not the resolved form), audit-trio (`updated_at` / `source`), and `invalidateUserTxCache` invariants are unchanged. **Stdio surface narrower by design (unchanged).** Stdio still rejects `portfolioHolding` / `portfolioHoldingId` / `quantity` at strict-mode validation; the new shape only adds `requestedValue` to the keys that branch on stdio (just `category`). Tool surface unchanged at 82 HTTP / 78 stdio. Files: [mcp-server/register-tools-pg.ts](mcp-server/register-tools-pg.ts), [mcp-server/register-core-tools.ts](mcp-server/register-core-tools.ts), [docs/architecture/mcp.md](docs/architecture/mcp.md).

### Added
- **MCP `bulk_record_transactions`: post-insert duplicate hints (2026-05-01, closes [#90](https://github.com/finlynq/finlynq/issues/90)).** After a successful (non-`dryRun`) `bulk_record_transactions` batch the response now carries a top-level `possibleDuplicates: PossibleDup[]` field flagging newly-inserted rows that look suspiciously like an existing row in the same account — same direction, amount within 5%, dates within 7 days. **Hints only — never blocks the insert.** Always present on success; empty array when nothing matches; never `null`. Skipped on `dryRun: true` (nothing inserted, nothing to scan against). **Match criteria, per inserted row:** same `account_id`, same direction (sign of amount; zero-amount rows skipped because direction is undefined for RSU vests / in-kind transfers), `|new.amount - existing.amount| / max(|new.amount|, |existing.amount|) <= 0.05`, `|new.date - existing.date| <= 7 days`, `existing.id != newId`. **Score** = `1 - (ratio/0.05) * 0.5 - (deltaDays/7) * 0.5` (range [0, 1], higher = closer match). The hint engine doesn't dedupe its own output — same `import_hash` pairs still surface so callers can inspect both legs. **HTTP path** ([mcp-server/register-tools-pg.ts](mcp-server/register-tools-pg.ts)) collects every committed row via `RETURNING id`, then runs ONE indexed pool query bounded by `[globalMinDate-7d, globalMaxDate+7d]` across every account that received a row, then filters per-row in JS through the shared scoring helper. Existing-row payees are decrypted via `tryDecryptField(dek, payee_raw, "transactions.payee") ?? rawPayee` per CLAUDE.md "tryDecryptField MUST return null on auth-tag failure" — auth-tag-failed entries decode to empty string rather than the v1: ciphertext, preserving the `?? plaintext` fallback at every call site. **Stdio path** ([mcp-server/register-core-tools.ts](mcp-server/register-core-tools.ts)) follows the same shape but reads payees as plaintext (stdio writes are plaintext per the existing stdio carve-out). Stdio's prepared INSERT statement gains `RETURNING id` and reads `runRes.lastInsertRowid` — the pg-compat shim's `run()` already plumbs `RETURNING id` rows to `lastInsertRowid` so the change is a one-line addition. **Shared helper** at [src/lib/mcp/duplicate-hints.ts](src/lib/mcp/duplicate-hints.ts) — `scanForPossibleDuplicates(committed, candidates)` returns `PossibleDup[]`; `dateBoundsForScan(committed)` computes the union date window. Pure functions, no DB / encryption / I/O dependencies. Both transports build their own candidate pool (encrypted vs plaintext payee handling differs) and feed the shared scoring helper — duplication is explicit and labelled with a "MIRROR" comment so future drift is obvious. **Performance.** Worst case: a 100-row batch into one account → one `SELECT ... WHERE user_id = ? AND account_id = ANY(...) AND date BETWEEN ?-7d AND ?+7d AND id <> ALL(<newIds>)`, indexed on `(user_id, account_id, date)`, fine. Across many accounts in one batch the `ANY($accountIds)` form keeps it to one query. **Scan must never fail the response** — wrapped in try/catch; on error it logs `[bulk_record_transactions] duplicate-hint scan failed:` and returns an empty `possibleDuplicates: []` so the caller still sees `imported` / `failed` / `results`. **`invalidateUserTxCache(userId)` ordering preserved** — invalidation still fires before the scan so a scan exception can't leave the cache stale. **Audit trio (`source` / `created_at` / `updated_at`) untouched** — purely additive read on the existing rows; no UPDATE, no `source` mutation. **Tool surface unchanged at 82 HTTP / 78 stdio** — purely additive on the response shape; no schema migration, no env vars, no cron, no MCP tool count change. Tool descriptions updated symmetrically on both transports. Files: [src/lib/mcp/duplicate-hints.ts](src/lib/mcp/duplicate-hints.ts), [mcp-server/register-tools-pg.ts](mcp-server/register-tools-pg.ts), [mcp-server/register-core-tools.ts](mcp-server/register-core-tools.ts), [docs/architecture/mcp.md](docs/architecture/mcp.md).

- **Cross-tenant FK leak in backup-restore + atomic, strict wipe (2026-04-27).** Two commits ([`5571070`](https://github.com/finlynq/finlynq/commit/5571070) → [`ad87419`](https://github.com/finlynq/finlynq/commit/ad87419)) closing one root cause and one wipe-side hardening, surfaced by the admin's "Wipe Account" 500ing with `Failed query: delete from "accounts" where "accounts"."user_id" = $1`.

  **Root cause — `/api/data/import` strip().** The JSON backup-restore handler at [src/app/api/data/import/route.ts](src/app/api/data/import/route.ts) used a generic `strip(rows, userId)` helper that only deleted `id` and forced `userId`, leaving `accountId` / `categoryId` / `assignCategoryId` columns UNCHANGED from the source backup. The route already builds an `accountIdMap` and `categoryIdMap` (old-id → new-id, populated when accounts and categories are inserted), and `transactions` + `transaction_splits` had explicit per-table remap blocks — but the strip-based inserts for `portfolio_holdings`, `loans`, `goals`, `snapshots`, `recurring_transactions`, `subscriptions`, `budgets`, `budget_templates`, and `transaction_rules` didn't touch the maps. So restoring user A's backup as user B silently wrote rows owned by B but with FK references to A's account/category integer IDs — accepted by Postgres because those IDs still existed in the same DB, producing a cross-tenant data leak that also became a wipe-blocker via FK 23503 the next time A tried to delete their account. Fix: `strip()` now takes optional `{ accountIdMap, categoryIdMap }`, remaps the FK columns in-place, and **throws on an unmapped FK** rather than passing the raw id through (fail-loud — a corrupt or partial backup is now a 500 instead of a silent leak). Every restore call site that touches an FK-bearing table passes the right map; tables with no FKs keep the old single-arg call. Concrete prod blast: a test user `pathfinder` restored the admin's backup, which created 60 portfolio_holdings owned by `pathfinder` but pointing at admin's accounts (IDs 437–461). All 60 rows were cleaned up via SQL as part of the same fix — they had no business existing.

  **Wipe-side hardening — atomicity + strict isolation.** [`wipeUserDataAndRewrap`](src/lib/auth/queries.ts) used to run each `db.delete(...)` as its own auto-commit, so a late failure (like the FK violation above) left the user signed in to a half-wiped account whose DEK was never rotated, and re-running the wipe just looped on the same FK. Wrap the entire delete sequence + the final DEK rewrap in a single `db.transaction(async (tx) => ...)` — atomic. Every delete still filters strictly by `user_id` only — **never by FK reach** — so the wipe can ONLY remove rows owned by the wiping user. If a cross-tenant row exists, the wipe fails with FK 23503 + transaction rollback rather than silently destroying another user's data. (An earlier version of `5571070` did add a defensive `account_id`-keyed sweep across all 8 referrer tables; that was reverted in `ad87419` because the right fix is to prevent cross-tenant rows from being created in the first place — the `strip()` change above.) The `mcp_uploads` file unlink loop runs BEFORE the DB transaction so we'd rather leak a DB row than orphan a plaintext file on disk if the wipe later fails.

- **Decryption resilience for stale-DEK ciphertext (2026-04-27).** Three-commit batch ([`152e4e6`](https://github.com/finlynq/finlynq/commit/152e4e6) → [`b45c9cf`](https://github.com/finlynq/finlynq/commit/b45c9cf) → [`efd7de2`](https://github.com/finlynq/finlynq/commit/efd7de2)) that hardens every read path against AES-GCM auth-tag failures so a single undecryptable row no longer 500s the whole endpoint. Triggered by a prod incident on user `pathfinder` whose `name_ct` ciphertext columns can't be decrypted with the current cached DEK (root cause still under investigation — scheduled agent to dig in).

  **Soft-fallback decryption.** [decryptName](src/lib/crypto/encrypted-columns.ts), [decryptTxRow](src/lib/crypto/encrypted-columns.ts), and [decryptSplitRow](src/lib/crypto/encrypted-columns.ts) now `try/catch` `Decipheriv.final()`. On failure they log a single `[envelope] decryptName failed` warn (so the underlying mismatch stays observable) and return the plaintext column when present, or the raw `v1:...` ciphertext as a UI marker when there's no fallback. Encryption is defense-in-depth — when the dual-write plaintext is still on disk, rendering it beats crashing the dashboard.

  **`tryDecryptField` helper** added in [envelope.ts](src/lib/crypto/envelope.ts). Same as `decryptField` but returns `null` (not the raw ciphertext, not a throw) on auth-tag failure, so callers can use the standard `tryDecryptField(...) ?? plaintextFallback` pattern.   **Primary: ZIP export upload.** User downloads WealthPosition → Profile → Export (a 4-CSV bundle: Accounts.csv / Categories.csv / Portfolio.csv / Transactions.csv) and uploads the ZIP. Server extracts via `jszip` (no native deps), parses via a hand-rolled RFC-4180 CSV reader ([csv.ts](packages/import-connectors/src/wealthposition/csv.ts)), and transforms into Finlynq's `RawTransaction[]` + `transaction_splits[]` + `portfolio_holdings[]`. Routes: [`/api/import/connectors/wealthposition/zip-{probe,preview,execute}`](src/app/api/import/connectors/wealthposition/). Instant (no rate limits) and handles 100% of transaction shapes. Portfolio.csv is the key: it's the authoritative holding → brokerage + ticker symbol mapping, which WP's API never exposes. Verified against a real 6,596-row export → 5,817 flat txs + 111 split parents + 60 portfolio_holdings rows + 0 transform errors.

  **Secondary: API key.** Live [WP API](https://www.wealthposition.com/api/v0.1) access, kept for balance reconciliation against `/account_balances`. Key is AES-GCM-encrypted under the user's DEK and persisted in `settings` under `connector:wealthposition:credentials`. Rate-limited client (1 req/s) with SHA-keyed global bucket on `globalThis` so state survives across HTTP requests in the same process; automatic retry with jittered backoff on `RATE_LIMIT_ERROR` / HTTP 429. Original plan had the API as the primary import path — pivoted to ZIP after discovering the API's `/accounts` flattens every position (Bitcoin, Ethereum, every stock ticker) as a separate top-level account with no way to tie them back to their brokerage.

  **Transform rules (ZIP path).** The CSV uses a `#SPLIT#` sentinel to preserve multi-leg transactions (parent row has `account=<real>` + `categorization=#SPLIT#`, consecutive children have `account=#SPLIT#`). Walker groups parent + consecutive children, classifies each child as category / account / holding, and emits:
    - **Simple 1A + 1C** → 1 Finlynq flat tx (4,204 rows, 63.5%).
    - **Cash + #SPLIT# + N categories** → 1 parent + N rows in `transaction_splits` (paycheck splits).
    - **Cash + #SPLIT# + N holdings** → parent cash leg preserved on the funding account + N position legs routed via `Portfolio.csv` to each holding's own brokerage (so a stock buy funded from RBC lands the cash leg on RBC and the shares on Questrade - RESP, each with `portfolioHolding = holding_name` + `quantity`).
    - **Cash + #SPLIT# + N accounts** → transfer set (parent + N flat txs, all tagged with the Transfer category).
    - **Orphan #SPLIT# rows** → surfaced as transform errors (shown in the preview dialog, not dropped silently).

  **Mapping + reconciliation.** Probe returns Finlynq's live accounts + categories so the mapping dialog can render names (not opaque ids); stale overrides (pointing at accounts deleted outside the import flow) are filtered client-side before opening the dialog and re-validated server-side in `materializeZipMapping`. Reconciliation endpoint fetches WP's `/accounts` once to build a `name→UUID` bridge, so CSV synthetic ids (`csv:acct:<name>`) match WP's real UUIDs from `/account_balances`. After a successful import, reconciliation runs inline when an API key is on file — per-account WP-vs-Finlynq diff table surfaces in the summary card, with an "Add opening balance" affordance for constant-offset mismatches.
